> 目标读者：不会系统做前端设计、但想用 AI 快速复刻一个 Blog 网站结构的新手  
> 核心原则：借鉴结构和方法，不照搬品牌、文案、Logo、图片和专有内容。

## 1. 这套流程适合做什么

这套方法适合快速做出这些页面：

- Blog 首页
- 内容中心
- SaaS 文章列表页
- 行业洞察站
- 知识库首页
- 报告和白皮书列表页
- 产品更新日志页
- 轻量内容型官网

如果你只是想做一个能看的静态页面，可以做到。  
如果你想继续升级成 Nuxt、Next、Astro、Vue 或 React 项目，这套流程也能继续往下接。

## 2. 你需要准备什么

### 2.1 工具

建议准备这些工具：

| 工具 | 用途 |
|---|---|
| OpenCode | 读取本地文件、生成设计文档、对比代码、整合最终方案 |
| Big Pickle OpenCode Zen | OpenCode 内可用的免费模型，用来做分析和规划 |
| Google Stitch | 根据设计文档生成视觉更完整的静态 HTML/CSS 页面 |
| Gemini | 根据设计文档生成更偏交互和路由的页面 |
| 浏览器 | 打开参考网站、检查生成结果 |
| VS Code 或任意编辑器 | 保存和查看 Markdown、HTML、CSS 文件 |

### 2.2 参考网站

本文以这个 Blog 网站作为参考对象：

```text
https://blog.insightrackr.com/?referer=ist_blog
```

你也可以换成任何你喜欢的 Blog 网站。

### 2.3 推荐文件结构

新手最容易乱在“文件到处放”。建议一开始就建立清晰目录：

```text
blog-copy-project/
├── 00-reference/
│   ├── screenshots/
│   └── page-notes.md
├── 01-analysis/
│   └── reference-analysis.md
├── 02-design/
│   ├── frontend-design-plan.md
│   └── blog-design-simple.md
├── 03-prototypes/
│   ├── copy_blog_stitch.html
│   └── copy_blog.html
├── 04-system/
│   ├── acceptance-checklist.md
│   └── implementation-plan.md
└── 05-final/
    └── final-blog.html
```

如果你只想简单做，也可以先把文件都放在同一个文件夹里。  
但只要文件多起来，目录结构会救你一命。

## 3. 总流程：不要一上来就让 AI 写代码

很多新手一开始会这样问：

```text
帮我模仿这个网站写一个 HTML 页面。
```

这种问法能得到页面，但质量经常不稳定。更好的方法是先做设计规格，再做代码。

推荐流程：

```text
第一步：放大分析参考站
第二步：策划完整设计方案
第三步：压缩成小白可执行 MVP
第四步：加密核心细节
第五步：生成验收清单
第六步：用 Stitch 生成视觉原型
第七步：用 Gemini 生成交互原型
第八步：让 OpenCode 对比融合
第九步：生成最终实现
第十步：按验收清单检查
```

这套流程看起来比“直接写代码”多几步，但实际上更稳。因为你先把需求讲清楚了，后面模型犯错会少很多。

## 4. 第一步：放大分析参考站

### 4.1 这一步的目的

这一步不要写代码。  
目标是让 AI 把参考网站拆开，看清楚它由哪些部分组成。

你要得到的是：

- 信息架构
- 页面布局
- 具体文案结构
- 颜色、字体、间距、圆角、阴影
- Header、卡片、标签、侧边栏、搜索、分页、Footer
- 桌面、平板、手机的布局变化
- 可复用设计规律

### 4.2 给 OpenCode 的提示词

打开 OpenCode，选择免费模型 `Big Pickle OpenCode Zen`，输入：

```text
请分析这个 Blog 页面：

https://blog.insightrackr.com/?referer=ist_blog

请从以下维度拆解，不要写代码，只输出可复用的设计和实现规律：

1. 信息架构和具体文案
   - 页面有哪些区域
   - 每个区域承担什么作用
   - 标题、摘要、分类、日期、按钮等文案如何组织

2. 布局结构
   - 请用 ASCII 图说明桌面端结构
   - 请说明侧边栏、主内容区、文章卡片区域的位置关系
   - 请说明各区域大致宽度、间距和层级

3. 设计系统
   - 颜色
   - 字体
   - 字号
   - 间距
   - 圆角
   - 阴影
   - 分割线
   - 背景色

4. 组件规范
   - Header
   - 搜索框
   - 侧边栏
   - 分类标签
   - 文章卡片
   - 分页器
   - Footer
   - 二维码或社交入口

5. 响应式策略
   - 桌面端如何布局
   - 平板端如何变化
   - 手机端如何变化
   - 侧边栏、搜索、卡片列数、分页在不同屏幕下如何处理

6. 内容模型
   - 每篇文章需要哪些字段
   - 分类和标签如何设计
   - 哪些字段用于搜索
   - 哪些字段用于排序和分页

7. SEO 和可访问性
   - 标题层级如何设计
   - 列表页如何利于搜索引擎收录
   - 按钮、链接、输入框需要哪些基本可访问性处理

请保存为 Markdown 文件：reference-analysis.md
```

### 4.3 你应该检查什么

模型生成后，不要急着进入下一步。先看 `reference-analysis.md` 是否包含这些内容：

- 有没有 ASCII 布局图
- 有没有组件清单
- 有没有颜色、字体、间距、圆角
- 有没有桌面、平板、手机说明
- 有没有内容模型
- 有没有 SEO 或可访问性提醒

如果缺了，就继续追问：

```text
请补充缺失部分，重点补充组件规范、内容模型和三端响应式策略。
```

## 5. 第二步：策划完整设计方案

### 5.1 这一步的目的

第一步是分析别人。  
第二步是做你自己的新方案。

注意：不是让 AI 继续照搬原站，而是让它模仿参考站的结构规律，策划一个新的 Blog 前端设计方案。

### 5.2 给 OpenCode 的提示词

```text
请基于 reference-analysis.md 的分析，策划一个新的类似 Blog 前端设计方案。

要求：

1. 不要照搬原站品牌、Logo、文案和图片。
2. 保留参考站的结构优点，例如：
   - 清晰 Header
   - 侧边分类
   - 文章卡片网格
   - 搜索入口
   - 分页
   - Footer
3. 输出一个完整设计方案，包含：
   - 信息架构
   - 页面路由
   - 首页布局
   - 文章详情页布局
   - 分类页
   - 搜索页
   - 404 页
   - 设计 token
   - 组件规范
   - 响应式策略
   - 技术栈建议
   - 关键设计决策和理由

请保存为 Markdown 文件：frontend-design-plan.md
```

### 5.3 这一步常见输出

模型通常会生成比较完整的方案，可能包含：

- 首页
- 详情页
- 分类页
- 搜索页
- 404
- Header
- 侧边栏
- 文章卡片
- Tag
- 分页器
- 搜索
- Footer
- 骨架屏
- Nuxt 或 Next 技术方案

这一步可能会显得复杂。不要慌。复杂是正常的，因为我们正在“放大”。

## 6. 第三步：压缩成小白可执行 MVP

### 6.1 这一步的目的

完整方案很有用，但新手直接照着做会压力很大。  
所以第三步要把它压缩成一个最小可执行版本。

MVP 的意思是：先做最核心、最能跑起来的一版。

对于 Blog 首页，MVP 只需要这些：

- Header
- 搜索框
- 分类侧边栏
- 文章卡片列表
- 分页器
- Footer
- 手机端布局

暂时可以不要：

- 复杂详情页
- 多语言
- 登录
- 评论系统
- 骨架屏
- 后台 CMS
- 复杂动画

### 6.2 给 OpenCode 的提示词

```text
frontend-design-plan.md 太复杂了。

请把它压缩成一个适合小白第一版实现的 Blog 首页 MVP 方案。

只保留必须实现的内容：

1. Header
2. 搜索框
3. 分类侧边栏
4. 文章卡片网格
5. 分页器
6. Footer
7. 桌面、平板、手机响应式布局

请删除或暂缓：

1. 复杂文章详情页
2. 骨架屏
3. 多语言
4. 登录注册
5. 评论系统
6. 后台管理
7. 复杂路由

请输出一个简化但仍然完整的设计文档，保存为：

blog-design-simple.md
```

### 6.3 小白判断标准

压缩后的文档应该让你感觉：

```text
我看得懂；
我知道要做哪些组件；
我知道页面大概长什么样；
我知道桌面和手机有什么区别；
我能拿它去让别的 AI 生成 HTML/CSS。
```

如果模型压缩得太狠，内容变得很空，就继续补一句：

```text
这个版本太简单了，请保留必要的布局尺寸、组件状态和响应式说明。
```

## 7. 第四步：加密核心细节

### 7.1 为什么要加密

新手常犯的错误是：文档看起来有设计，但代码生成出来很松散。

原因通常是关键细节不够密：

- 卡片多宽
- 图片什么比例
- 标题最多几行
- 摘要最多几行
- hover 怎么动
- 手机端几列
- 侧边栏怎么折叠
- 搜索框在哪
- 分页在手机上怎么显示

所以第四步不是把所有内容都细化，而是只细化最容易翻车的部分。

### 7.2 给 OpenCode 的提示词

```text
请继续细化 blog-design-simple.md。

不要扩展新功能，只细化以下高风险部分：

1. 布局结构
   - 给出桌面端 ASCII 布局图
   - 给出平板端 ASCII 布局图
   - 给出手机端 ASCII 布局图
   - 标注 Header 高度、侧边栏宽度、内容区最大宽度、卡片间距

2. 设计 token
   - 用 CSS 自定义属性格式输出
   - 包含颜色、字体、字号、行高、间距、圆角、阴影、断点

3. 文章卡片细节
   - 卡片宽度和高度建议
   - 图片比例
   - 标签样式
   - 标题最多几行
   - 摘要最多几行
   - 日期位置
   - hover 动效
   - 阴影变化
   - 内边距

4. 组件规范
   - Header
   - 搜索框
   - 侧边栏
   - 分类按钮
   - 文章卡片
   - 分页器
   - Footer

5. 响应式行为
   - 桌面端列数
   - 平板端列数
   - 手机端列数
   - 侧边栏在不同屏幕下如何变化
   - 搜索框在不同屏幕下如何变化
   - 分页在手机端如何简化

请直接更新 blog-design-simple.md。
```

### 7.3 最好得到什么结果

这一步之后，`blog-design-simple.md` 应该变成可以直接喂给 Stitch 或 Gemini 的规格文档。

它应该包含类似这样的内容：

```css
:root {
  --color-bg: #f8fafc;
  --color-surface: #ffffff;
  --color-text: #111827;
  --color-muted: #6b7280;
  --radius-card: 8px;
  --space-4: 16px;
  --shadow-card: 0 4px 12px rgba(15, 23, 42, 0.08);
}
```

还应该包含类似这样的卡片规则：

```text
文章卡片：
- 桌面端 3 列
- 平板端 2 列
- 手机端 1 列
- 封面图比例 16:9
- 标题最多 2 行
- 摘要最多 2 行
- hover 时向上移动 2px
- hover 时阴影加深
```

## 8. 第五步：生成验收清单

### 8.1 这一步很重要

原始流程里没有这一环，但强烈建议补上。

因为 AI 生成页面时，经常会出现这种问题：

- 看起来像，但移动端乱了
- 有搜索框，但不能用
- 卡片好看，但数据写死
- 有分页，但只是装饰
- Header 在小屏幕挤爆
- 颜色不统一
- 文案和参考站太像

所以在生成代码前，先让 AI 生成验收清单。

### 8.2 给 OpenCode 的提示词

```text
请基于 blog-design-simple.md 生成一份实现验收清单。

要求覆盖以下维度：

1. 视觉一致性
2. 组件完整性
3. 桌面、平板、手机响应式
4. 搜索、分类、分页交互
5. 文章数据结构
6. SEO 基础
7. 可访问性基础
8. 性能基础
9. 版权和品牌边界

每一项都写成可以检查的句子，例如：

- 桌面端文章列表为 3 列网格；
- 手机端文章列表为单列；
- 文章卡片标题最多显示 2 行；
- 搜索应匹配标题、摘要、分类和标签；
- 不使用参考站原始 Logo、图片和文章全文。

请保存为 Markdown 文件：acceptance-checklist.md
```

### 8.3 验收清单示例

你最终可以用下面这类问题检查页面：

| 维度 | 检查问题 |
|---|---|
| 视觉 | 页面是否有统一颜色、圆角、阴影和间距 |
| Header | Logo、搜索、导航入口是否清楚 |
| 侧边栏 | 分类是否可点击，选中态是否明显 |
| 文章卡片 | 图片、标签、标题、摘要、日期是否完整 |
| 搜索 | 是否能按标题、摘要、分类、标签过滤 |
| 分页 | 是否能切换页码，手机端是否不拥挤 |
| 响应式 | 桌面 3 列、平板 2 列、手机 1 列是否成立 |
| SEO | 是否有语义化标题和文章列表结构 |
| 版权 | 是否避免照搬原站品牌资产 |

## 9. 第六步：用 Stitch 生成视觉原型

### 9.1 这一步的目的

Stitch 比较适合生成页面视觉细节。  
它通常能做出比较完整、比较像设计稿的静态页面。

缺点是：交互、路由、数据逻辑可能比较弱。

所以我们把 Stitch 生成的结果当作“视觉样板”。

### 9.2 操作方法

打开：

```text
https://stitch.withgoogle.com/?pli=1
```

上传：

```text
blog-design-simple.md
```

然后输入：

```text
请根据我上传的 blog-design-simple.md，用 HTML + CSS 实现一个静态 Blog 首页。

要求：

1. 尽量严格遵守文档里的布局、token、组件和卡片细节；
2. 页面只需要静态展示，不需要复杂交互；
3. 重点保证视觉完成度；
4. 必须包含 Header、搜索框、侧边栏、文章卡片、分页器、Footer；
5. 必须包含桌面、平板、手机响应式；
6. 不要使用参考站的 Logo、品牌名、原文案和原图片；
7. 文章数据可以使用虚构示例内容。

请输出完整 HTML + CSS 代码。
```

生成后，把代码保存为：

```text
copy_blog_stitch.html
```

### 9.3 检查重点

打开 `copy_blog_stitch.html`，重点看：

- 页面是否好看
- 卡片是否统一
- 间距是否舒服
- 字体层级是否清楚
- 手机端是否正常
- 是否有明显视觉问题

如果视觉不错，即使没交互也没关系。它的角色就是视觉样板。

## 10. 第七步：用 Gemini 生成交互原型

### 10.1 这一步的目的

Gemini 生成的页面视觉可能不如 Stitch 稳，但经常更愿意写一点交互逻辑。

比如：

- 搜索过滤
- 分类筛选
- 分页切换
- 简单路由模拟
- 卡片点击详情
- 移动菜单

所以我们把 Gemini 生成的结果当作“行为样板”。

### 10.2 操作方法

打开：

```text
https://gemini.google.com/
```

上传：

```text
blog-design-simple.md
```

然后输入：

```text
请根据我上传的 blog-design-simple.md，用 HTML + CSS + JavaScript 实现一个 Blog 首页原型。

要求：

1. 页面包含 Header、搜索框、侧边栏、文章卡片、分页器、Footer；
2. 使用虚构文章数据数组驱动页面；
3. 搜索可以匹配标题、摘要、分类和标签；
4. 分类点击后可以筛选文章；
5. 分页可以切换；
6. 移动端布局可用；
7. 视觉尽量遵守文档里的 token 和卡片规范；
8. 不要使用参考站的 Logo、品牌名、原文案和原图片。

请输出一个完整可运行的 HTML 文件。
```

生成后，把代码保存为：

```text
copy_blog.html
```

### 10.3 检查重点

打开 `copy_blog.html`，重点看：

- 搜索是否能用
- 分类是否能筛选
- 分页是否能切换
- 文章数据是否集中管理
- 手机端菜单是否可用
- 点击卡片是否有反馈

即使视觉不完美也没关系。它的角色是行为样板。

## 11. 第八步：让 OpenCode 对比融合两个原型

### 11.1 这一步的目的

现在你手里有两个原型：

```text
copy_blog_stitch.html：视觉更好
copy_blog.html：交互更好
```

接下来不要手工乱抄。  
让 OpenCode 对两份代码做系统对比，然后融合出最佳实现方案。

### 11.2 给 OpenCode 的提示词

```text
请读取以下文件：

1. frontend-design-plan.md
2. blog-design-simple.md
3. acceptance-checklist.md
4. copy_blog_stitch.html
5. copy_blog.html

请完成一份完整系统的最佳实现方案，保存为：

implementation-plan.md

要求：

1. 逐维度对比 copy_blog_stitch.html 和 copy_blog.html
   - 视觉完成度
   - 布局结构
   - 组件完整性
   - 搜索交互
   - 分类筛选
   - 分页
   - 移动端
   - 代码结构
   - 可维护性

2. 提取两份文件各自最佳实践
   - Stitch 版本有哪些视觉细节值得保留
   - Gemini 版本有哪些交互逻辑值得保留

3. 融合出一套完整组件体系
   - Header
   - 侧边栏
```

## 13. 第十步：按验收清单检查


## 14. 常见问题

### 问题 1：为什么不直接让 AI 写 HTML

可以直接写，但容易得到一次性页面。  
本文的方法先生成 Markdown 规格，是为了让每一步都可检查、可修改、可复用。

简单说：

```text
HTML 是结果；
Markdown 规格是方法；
方法比一次结果更值钱。
```

### 问题 2：为什么要同时用 Stitch 和 Gemini

因为它们擅长的方向不同。

```text
Stitch 更像视觉设计师；
Gemini 更像交互原型师；
OpenCode 更像前端架构师。
```

让不同工具做自己擅长的事，比期待一个工具全能更稳。

### 问题 3：如果 Stitch 生成的页面没有交互怎么办

没关系。  
Stitch 的任务是提供视觉样板，不是完成全部功能。

后面可以让 OpenCode 把 Gemini 的交互逻辑融合进去。

### 问题 4：如果 Gemini 生成的页面不好看怎么办

也没关系。  
Gemini 的任务是提供搜索、分类、分页等行为样板。

最终视觉以 Stitch 或你自己的设计文档为准。

### 问题 5：如果两个原型代码风格完全不同怎么办

不要强行拼接。  
让 OpenCode 先写 `implementation-plan.md`，把两份代码拆成：

- 哪些视觉值得保留
- 哪些交互值得保留
- 哪些代码需要重写
- 哪些功能暂缓

然后重新生成最终页面。

## 17. 结论

这套方法的关键不是“会不会写代码”，而是“会不会把网站拆成规格”。

最重要的顺序是：

```text
先放大：完整分析参考站，避免遗漏；
再压缩：变成小白能落地的 MVP；
再加密：细化最容易翻车的布局、token、卡片和响应式；
再验收：生成清单，防止页面只是看起来像；
最后实现：用不同模型互补，再让 OpenCode 融合。
```

如果你只记住一句话，就是：

```text
不要直接问 AI 要最终代码；
先让 AI 生成可审查的设计规格；
再用不同模型生成互补原型；
最后对比融合，生成自己的最终页面。
```
