## 多工具协作框架

### 1. Gemini + GPT：框架思路与代码骨架

**角色定位：** 高层架构师、方案规划者

**应用场景：**
- 快速原型开发（Prompt → 可运行代码）
- 多语言全栈开发（前端/后端/移动端/数据库脚本一键生成）
- 复杂系统重构与长期工程任务

**实践心得：**
- 用Gemini进行方案讨论、框架思路生成
- GPT-5.5处理长时程多步骤任务，能自主规划、用工具、检查输出、迭代修复
- 两者结合可以在项目初期快速产出代码骨架和架构文档

**关键配置：**
```markdown
# 项目启动Prompt示例
## 目标
构建一个xxx系统

## 非目标
- 暂不考虑xxx
- 排除xxx功能

## 硬约束
- 使用xxx技术栈
- 遵循xxx规范

## 交付物
- 完整项目代码
- 架构文档
- 部署指南

## Done When
- 所有单元测试通过
- 核心功能可正常运行
```

### 2. NotebookLM：矛盾点分析与盲点检测

**角色定位：** 质量审计员、思维补全者

**应用场景：**
- 将项目文档、代码、设计稿上传作为知识源
- 要求AI进行矛盾点分析、盲点检测
- 整合系统组织框架和线索

**实践流程：**
1. 将Gemini/GPT产出的代码骨架和文档上传
2. 提问："请分析这套方案的潜在矛盾点和设计盲点"
3. 要求输出："整合所有分析结果，给出改进建议清单"
4. 将改进后的方案再次输入，形成闭环验证

### 3. Codex：小白导师与任务拆解

**角色定位：** 技术教练、任务分解师

**应用场景：**
- 指导手动操作命令
- 任务拆解与细化
- 步骤式教学

**Codex的自主性特征：**
- 处理长时程、多步骤任务
- 自主规划、用工具、检查输出、迭代修复
- 对高层次指令响应极强
- 能处理"messy, multi-part task"，自己完成plan → execute → check → iterate循环

**任务拆解示例：**
```markdown
# 复杂任务拆解框架

## Milestone 1: 环境搭建
- [ ] 安装依赖
- [ ] 配置开发环境
- [ ] 验证基础功能

## Milestone 2: 核心模块
- [ ] 数据模型设计
- [ ] API接口开发
- [ ] 单元测试编写

## Milestone 3: 集成测试
- [ ] 模块间联调
- [ ] 性能测试
- [ ] Bug修复

每个Milestone包含：
- Acceptance Criteria（验收标准）
- Validation Commands（验证命令）
```

### 4. Continue：深度咨询与技术细化

**角色定位：** 本地技术顾问、实时编程伙伴

**四大核心模式：**
| 模式 | 快捷键 | 适用场景 |
|------|--------|----------|
| **Chat** | Ctrl+L | 引用项目文件/终端输出，分析问题 |
| **Edit** | Ctrl+I | 选中代码直接修改，精准重构 |
| **Autocomplete** | Tab | 实时代码补全 |
| **Agent** | - | 多文件任务自动化执行 |

**将Codex指导保存为MD的工作流：**
1. Codex输出任务拆解和指导内容 → 保存为`.md`文件
2. 在Continue中使用`@file`引用该MD文件
3. 基于Codex的指导，用Chat模式咨询更细的技术问题
4. 用Edit模式直接修改代码，用Agent模式执行多文件任务

**Context Provider快速调用：**
- `@file` - 引用任意文件
- `@terminal` - 引用终端报错
- `@codebase` - 搜索整个项目相关代码
- `@docs` - 引用外部文档

### 5. OpenCode + Continue + Codex：脚本生成与审查闭环

**角色定位：** 代码生成、上下文摘录、质量审查的三重协作

**适用场景：**
- 需要编写具体脚本代码
- 需要结合项目上下文环境
- 需要多轮审查优化

#### 完整工作流程

```
┌─────────────────────────────────────────────────────────────────────┐
│              OpenCode → Continue → Codex 脚本协作闭环               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Step 1: OpenCode + MiMo-V2.5-Free                                 │
│  ────────────────────────────────────                               │
│  • 任务：编写具体脚本                                                │
│  • 输入：需求描述 + 技术约束                                         │
│  • 输出：初版代码脚本                                                │
│                                                                     │
│  Step 2: Continue 上下文摘录                                        │
│  ────────────────────────                                           │
│  • 操作：使用 @file / @codebase 引用项目现有代码                      │
│  • 任务：识别哪些脚本片段适合当前上下文                                │
│  • 输出：精选的代码片段 + 适配建议                                    │
│                                                                     │
│  Step 3: Codex 审查                                                 │
│  ────────────────                                                   │
│  • 任务：审查代码合理性                                               │
│  • 检查：逻辑是否正确、是否有优化空间、是否符合最佳实践                  │
│  • 输出：审查意见 + 优化建议                                          │
│                                                                     │
│  Step 4: 迭代                                                       │
│  ──────────                                                         │
│  • 将Codex反馈回到OpenCode/Continue进行修改                           │
│  • 形成闭环直到满意                                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### 详细操作步骤

**Step 1：OpenCode生成初版脚本**

```bash
# 启动OpenCode，选择MiMo-V2.5-Free模型
opencode

# 在终端中输入任务描述
> 请编写一个Python脚本，功能是：
> 1. 读取指定目录下的所有.md文件
> 2. 提取每个文件的frontmatter信息
> 3. 生成一份汇总索引文件
> 要求：使用pathlib，添加错误处理，包含类型注解
```

OpenCode会输出完整的脚本代码。

**Step 2：Continue进行上下文适配**

```markdown
# 在Continue中执行以下操作

## 1. 引用项目现有代码
@file: src/utils/file_handler.py  # 参考现有的文件处理方式
@codebase: 搜索项目中类似的解析逻辑

## 2. 提问
"OpenCode生成的脚本中，文件读取部分与项目现有的file_handler.py
风格不一致，请帮我摘录合适的片段并适配到当前项目上下文"

## 3. Continue输出
- 精选的代码片段（符合项目现有风格）
- 适配后的代码（保持功能一致，风格统一）
```

**Step 3：Codex进行合理性审查**

```markdown
# 将Continue适配后的代码交给Codex审查

## Codex审查指令
请审查以下Python脚本：
[粘贴适配后的代码]

## 审查要点
1. 逻辑正确性：是否能正确提取frontmatter
2. 错误处理：异常情况是否覆盖完整
3. 代码质量：是否符合Python最佳实践
4. 优化空间：是否有性能或可读性改进点

## Codex输出示例
### 审查结论：基本合理，建议优化
- ✅ 逻辑正确，能正确解析YAML frontmatter
- ⚠️ 建议：添加yaml库的导入检查
- ⚠️ 建议：文件编码应显式指定为utf-8
- 💡 优化：可使用异步IO提升大目录处理性能
```

#### 实战示例：数据分析脚本

```markdown
# 完整协作流程示例

## 原始需求
"编写一个脚本，自动分析CSV文件中的销售数据，
生成月度汇总报表"

## Step 1: OpenCode输出
```python
import pandas as pd
from pathlib import Path

def analyze_sales(csv_path: Path) -> dict:
    """分析销售数据并返回月度汇总"""
    df = pd.read_csv(csv_path)
    df['date'] = pd.to_datetime(df['date'])
    monthly = df.groupby(df['date'].dt.month)['amount'].sum()
    return monthly.to_dict()
```

## Step 2: Continue上下文摘录
@codebase: 搜索项目中pandas的使用方式

- ✅ 基础逻辑正确
- ⚠️ 缺少空文件处理
- ⚠️ 应使用项目统一的日志模块
- 💡 建议添加缓存机制避免重复计算

## 最终版本
[融合三方意见后的优化代码]
```

## 关键认知转变

### 从"写得出来"到"可控交付"

| 传统模式 | AI协作模式 |
|----------|------------|
| 手写全部代码 | 框架+骨架由AI生成 |
| 人工Code Review | AI预审+人工终审 |
| 手动测试 | AI自动化测试+人工验证 |
| 逐行Debug | AI分析根因+定向修复 |

### 人机分工新范式

**人负责：**
- 定义目标和验收标准
- 设计架构和技术选型
- 最终质量把关
- 处理模糊需求和边界情况

**AI负责：**
- 框架代码生成
- 重复性代码编写
- 自动化测试
- 代码审查和Bug定位
- 文档生成

## 实践建议

1. **给AI项目记忆：** 不要依赖单次Prompt，使用Markdown文件维护项目状态
2. **设计验收标准：** 明确"Done When"，让AI知道什么算完成
3. **建立证据链：** 记录AI的决策过程，便于审计和追溯
4. **设置熔断器：** 定义人工介入点，避免AI在错误方向上越跑越远
5. **保留恢复包：** 中断后能快速接续，不需要重新考古
